# 写入功能实施计划

本文面向后续执行实现的 AI，目标是把 SiYuan Agent Bridge 从只读工具扩展到第一阶段可控写入。实现时必须先阅读 `AGENTS.md`、`PRO.md`、`思源API.md`、`README.md` 和当前代码。

## 目标

第一阶段只实现两个面向 AI 的 MCP 写入工具：

1. `siyuan_create_document`
2. `siyuan_edit_document`

所有底层思源写 API 都必须由 MCP server 内部调用，不允许把 `updateBlock`、`appendBlock`、`deleteBlock`、`checkoutRepo` 等底层能力直接暴露给 AI。

核心目标：

- AI 用文档文本进行编辑，不直接操作块 ID。
- 每次写入前必须创建思源工作空间快照。
- 快照失败则拒绝写入。
- 第一阶段不做 AI 自动回滚；用户需要时在思源内手动回滚到快照。
- 写入完成后必须调用思源通知 API，让用户在 UI 中看到变更。
- 写入工具必须遵守现有隐私规则，隐藏文档不可写。

## 非目标

第一阶段不要实现：

- `siyuan_rollback`
- 独立的 `delete_block`
- 独立的 `update_block`
- 独立的 `insert_block`
- 独立的 `move_block`
- 文档重命名、移动、删除
- 笔记本创建、重命名、删除
- `checkoutRepo` 自动恢复
- 任意 SQL MCP 工具
- 图片/附件上传
- 跨文档块引用自动创建
- 整篇覆盖已有文档

## 已验证事实

当前分支已经做过备份 API 验证：

- 新工作空间端口可能不是默认 `6806`，本次测试使用过 `11965`。
- 不同工作空间可以有不同 token。本次使用的测试工作空间token为hlpse2zqsbaink1a
- `/api/repo/createSnapshot` 在未初始化数据仓库密钥时会失败，错误提示为需要先在 `设置 - 关于 - 数据仓库密钥` 初始化。
- 初始化密钥后，`createSnapshot` 成功时可能返回 `data: null`，不返回 snapshot id。
- 如果工作空间没有新的数据变更，`createSnapshot` 可能成功但不新增可见快照。
- 如果工作空间有新内容变更，`createSnapshot` 会在 `/api/repo/getRepoSnapshots` 中新增快照。
- 用户已验证可以手动回滚到本工具创建的快照状态。

这些行为已经记录在 `PRO.md` 和 `思源API.md`。

## API 能力清单

第一阶段需要封装的思源 API：

| API | 用途 |
|-----|------|
| `/api/repo/createSnapshot` | 写入前创建工作空间快照 |
| `/api/repo/getRepoSnapshots` | 开发诊断用，确认快照列表 |
| `/api/filetree/createDocWithMd` | 创建新文档 |
| `/api/block/updateBlock` | 替换单个块内容 |
| `/api/block/appendBlock` | 向文档末尾追加块 |
| `/api/block/insertBlock` | 在锚点块之后插入块 |
| `/api/query/sql` | 查询文档下块内容，用于文本锚点定位 |
| `/api/notification/pushMsg` | 写入成功通知 |
| `/api/notification/pushErrMsg` | 写入失败通知，可选 |

不要暴露这些底层 API 给 MCP 客户端。MCP 工具只能暴露高层语义。

## 命名决策

使用 `siyuan_edit_document`，不要使用 `siyuan_edit_block`。

原因：

- AI 的任务语义是“编辑文档中的文字”。
- 块只是思源内部实现细节。
- 与 Claude Code 的 Edit 心智模型一致：给出旧文本和新文本，而不是给出块 ID。

## 工具一：siyuan_create_document

### MCP 参数

```text
notebook_id: str
title: str
path: str | optional
markdown: str
confirmed: bool
```

### 语义

在指定笔记本/路径下创建一篇新文档。

### 服务端流程

1. 校验 `confirmed == true`。
2. 校验 `notebook_id` 存在，并且目标笔记本没有被隐私规则隐藏。
3. 校验 `title` 非空。
4. 校验 `markdown` 非空。
5. 组合目标路径。
6. 调用 `createSnapshot`。
7. 如果快照失败，拒绝创建文档，并返回明确错误：
   - 如果是数据仓库密钥未初始化，提示用户到思源 UI 初始化。
   - 其他错误直接返回。
8. 调用 `/api/filetree/createDocWithMd`。
9. 调用 `pushMsg`，提示创建完成。
10. 返回：
    - endpoint
    - notebook_id
    - title
    - path
    - document_id，如果 API 返回
    - snapshot_status
    - 用户可手动回滚的说明

### 注意

如果 `createDocWithMd` 不返回文档 ID，可以通过路径或 SQL 再解析一次。不要为了拿 ID 做过复杂的逻辑；第一阶段只要能创建并提示用户即可。

## 工具二：siyuan_edit_document

### MCP 参数

```text
document_id: str
old_text: str
new_text: str
confirmed: bool
```

### 语义

在已有可见文档内做局部编辑。

| old_text | new_text | 操作 |
|----------|----------|------|
| 非空 | 非空 | 替换 |
| 非空 | 空 | 删除匹配文本 |
| 空 | 非空 | 追加到文档末尾 |
| 非空 | 包含 old_text 且额外追加内容 | 可实现锚点后插入，第一阶段可先按替换处理 |

### 第一阶段范围

先只支持：

- 单块内完整替换
- 单块内子串替换
- 向文档末尾追加
- 删除单块内匹配文本

暂不支持：

- 跨块 old_text
- 同时修改多个块
- 移动块
- 文档结构重排
- 整篇覆盖

遇到跨块文本时，返回错误，让 AI 重新读取文档并拆成多次单块编辑。

### 服务端流程

1. 校验 `confirmed == true`。
2. 用现有 `resolve_visible_document` 或等价逻辑解析 `document_id`。
3. 如果文档被隐私规则隐藏，拒绝写入。
4. 如果 `old_text` 和 `new_text` 都为空，拒绝。
5. 如果 `old_text` 为空：
   - 调用 `createSnapshot`。
   - 调用 `appendBlock` 把 `new_text` 追加到文档末尾。
   - `pushMsg` 通知。
   - 返回结果。
6. 如果 `old_text` 非空：
   - 查询该文档下所有块的 Markdown/Kramdown 内容。
   - 在块级粒度搜索 `old_text`。
   - 如果 0 命中，拒绝，提示“文档可能已变化，请重新读取后提供原文”。
   - 如果多处命中，拒绝，返回少量上下文，让 AI 提供更长 `old_text`。
   - 如果唯一命中，继续。
7. 调用 `createSnapshot`。
8. 根据匹配形式执行：
   - `old_text == 块全文`：`updateBlock(block_id, new_text)`。
   - `old_text` 是块内子串：把块全文中的 `old_text` 替换为 `new_text`，再 `updateBlock`。
   - `new_text == ""`：按替换为空处理。
9. 调用 `pushMsg`。
10. 返回：
    - endpoint
    - document_id
    - document title/path
    - changed block count
    - snapshot_status
    - 简短 preview

## 写前快照要求

所有写入工具都必须先调用：

```text
/api/repo/createSnapshot
```

建议 memo 格式：

```text
siyuan-agent-bridge before <tool_name>: <target title/path> <timestamp>
```

标签建议：

```text
siyuan-agent-bridge
write
```

如果快照失败，不允许继续写入。

如果 `createSnapshot` 返回 `data: null`，视为请求成功。不要要求 snapshot id 必须存在。

实现时注意：如果当前工作空间没有未快照化变更，快照列表可能不会新增记录。这不代表 API 调用失败。

## 多工作空间连接问题

当前项目连接配置仍偏向单工作空间。写入功能会放大误连风险。

第一阶段实现时至少做到：

- 所有写入返回结果中包含当前 `client.base_url`。
- 错误信息中说明当前 endpoint。
- `siyuan_start` 或写入工具输出中尽量显示当前工作空间笔记本概览，方便用户确认。

暂不实现完整 profile 管理，但不要把端口或 token 写死。

后续可设计：

- workspace profile
- profile 对应 endpoint/token
- 用户在会话中选择当前工作空间
- 写入前确认 endpoint 和目标文档

## 隐私要求

写入工具必须沿用现有隐私模型：

- 隐藏笔记本不可写。
- 隐藏文档及其子文档不可写。
- 临时 allow 生效时，才允许写入对应文档。
- 不要为了写入绕过 `docs.jsonl` 和隐私规则。
- 如果要实时解析文档，需要同样套用 `filter_documents` / `rule_matches_live_doc` 等逻辑。

如果不能确认文档是否可见，拒绝写入。

## 客户端方法建议

在 `SiYuanClient` 中新增或保留：

```python
create_snapshot(memo: str, tags: list[str] | None = None) -> dict[str, Any]
get_repo_snapshots(page: int = 1) -> dict[str, Any]
create_doc_with_md(notebook_id: str, path: str, markdown: str) -> dict[str, Any]
update_block(block_id: str, markdown: str) -> dict[str, Any]
append_block(parent_id: str, markdown: str) -> dict[str, Any]
insert_block_after(previous_id: str, markdown: str) -> dict[str, Any]
push_msg(message: str) -> None
```

具体 payload 需要以思源 API 实测为准。不要猜测复杂字段；先写最小可用封装，再用测试固定行为。

## MCP 工具描述要求

工具描述必须明确：

- 会修改思源笔记。
- 必须 `confirmed=true`。
- 写入前会创建工作空间快照。
- 如果快照失败，不会写入。
- 回滚由用户在思源中手动完成。
- 只能编辑可见文档。

示例描述方向：

```text
Edit visible SiYuan document text using an exact old_text -> new_text anchor. Creates a SiYuan workspace snapshot before writing. Refuses ambiguous, missing, hidden, or unconfirmed edits.
```

## 测试计划

### 单元测试

新增测试覆盖：

- `create_snapshot` 接受 `data:null`。
- `create_doc_with_md` payload 正确。
- `update_block` payload 正确。
- `append_block` payload 正确。
- `push_msg` payload 正确。
- 写入工具 `confirmed=false` 拒绝。
- 隐藏文档拒绝写入。
- `old_text` 0 命中拒绝。
- `old_text` 多命中拒绝。
- 单块子串替换成功。
- `old_text=""` 追加成功。
- 快照失败时不调用写 API。

### 手动集成测试

在测试工作空间中执行：

1. 初始化数据仓库密钥。
2. 创建测试笔记本和测试文档。
3. 调用 `siyuan_create_document` 创建新文档。
4. 搜索/读取新文档确认内容存在。
5. 调用 `siyuan_edit_document` 替换一句话。
6. 搜索/读取确认替换成功。
7. 调用 `siyuan_edit_document` 追加一段。
8. 查询 `snapshots` 确认有写入前快照。
9. 用户手动回滚到快照。
10. 搜索/读取确认内容回到备份状态。

### 不要在真实主工作空间直接测试

写入能力实现早期只在测试工作空间验证。不要在用户主知识库上做破坏性测试。

## README / Skill 更新

实现写入工具后，必须更新用户文档和 skill。

当前 skill 仍写着：

```text
不要修改思源笔记内容。
不要调用思源写 API。
```

实现后应改为：

- 不要直接调用底层思源写 API。
- 只有在用户明确要求写入时，才使用 `siyuan_create_document` 或 `siyuan_edit_document`。
- 写入前必须说明将修改什么。
- 写入工具必须 `confirmed=true`。
- 不要尝试自动回滚。
- 如果用户要回滚，提示用户在思源快照中手动恢复。

README 也要从“只读工具”更新为“默认只读，明确确认后可写入”。

## 实施顺序

建议顺序：

1. 完善 `client.py` 写 API 封装。
2. 完善 CLI 诊断命令：`backup`、`snapshots`。
3. 实现 `siyuan_create_document`。
4. 为 `siyuan_create_document` 写测试。
5. 在测试工作空间手动验证创建文档和快照。
6. 实现 `siyuan_edit_document` 的追加和单块替换。
7. 为 `siyuan_edit_document` 写测试。
8. 在测试工作空间手动验证替换、追加、快照、手动回滚。
9. 更新 README、README.en、Skill。
10. 更新 `PRO.md` 的状态和实测结论。

不要先做复杂跨块编辑。第一阶段的价值是建立安全写入链路。

## 风险清单

| 风险 | 处理 |
|------|------|
| 连接到错误工作空间 | 写入结果显示 endpoint，未来做 profile |
| token 错误 | 保持现有连接错误提示 |
| 未初始化数据仓库密钥 | 快照失败，拒绝写入，提示用户初始化 |
| 快照成功但无 snapshot id | 接受 `data:null`，返回 snapshot_status |
| 快照列表未新增 | 记录为可能没有未快照化变更，不当作硬失败 |
| 文本锚点不唯一 | 拒绝，要求更长 old_text |
| 文档已被用户修改 | old_text 匹配失败，要求重新读取 |
| 隐私规则绕过 | 写入前必须 resolve visible document |
| 误删内容 | 删除只允许通过精确 old_text，且写前快照 |
| AI 自动回滚扩大损害 | 第一阶段不暴露 rollback/checkout |

## 完成标准

第一阶段完成时应满足：

- 两个 MCP 写入工具可用。
- 每次写入前自动调用 `createSnapshot`。
- 快照失败不会写入。
- 写入后思源 UI 有通知。
- 隐私隐藏内容不可写。
- 测试工作空间中创建、替换、追加、手动回滚全部验证通过。
- 文档和 Skill 已更新，不再声称项目绝对只读。
