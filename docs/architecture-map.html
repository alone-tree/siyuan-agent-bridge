目录地图

仓库里真正需要记住的不是每个文件，而是它们分别承担哪种边界。

Python 适配层

source_code/

mcp_server.py 是工具面和业务编排中心；client.py 封装思源 API；indexer.py 生成安全索引；ignore.py 解析隐私规则；agent_notebook.py 维护系统笔记本。

AI 插件层

plugins/

scripts/run_mcp.py 是 MCP stdio 启动脚本；两个 Skill 分别负责普通思源访问和 Workspace Index 构建。

运行时缓存

knowledge_base/

由刷新流程生成，包含客观文档树、结构化文档元数据、可见笔记本索引和隐私规则解析缓存。每次 refresh 可能覆盖。

AI 工作区

ai_workspace/

放附件提取、导出 Markdown、临时草稿和分析材料。只有 siyuan_start 会清理 README 外的内容。

工程文档

docs/

ARCHITECTURE.md 是真实架构和工具契约；DEVELOPMENT_GUIDE.md 是开发流程和验证规则；devlog.md 只放工程日志。

发布与配置

release

pack_skill.py、pack_release.py 打包发布；mcp_configs/ 放不同 AI 客户端的 MCP 配置模板。

关键流程

把最容易混淆的“启动、阅读、写入、搜索”拆成泳道。看流程时先看谁负责边界，再看数据去哪。

会话启动
siyuan_start→ detect profile→ ensure system notebook→ parse Privacy Rules→ refresh safe index→ startup packet
搜索
siyuan_find→ open closed notebooks temporarily→ SiYuan search / SQL→ privacy filter→ group snippets by doc
阅读
siyuan_read→ resolve visible doc→ getChildBlocks→ outline + block window→ extract attachments
写入
create / edit / doc_manage→ confirmed=true + permission→ create snapshot→ write SiYuan→ wait path sync→ safe refresh

系统笔记本生命周期

AI Guide 不存在时创建但不覆盖；Workspace Index 不自动创建；About 模板升级时覆盖；Privacy Rules 不覆盖且 AI 不可见。

客观索引与语义索引分工

程序生成 tree/docs/notebooks；AI 维护 Workspace Index；人维护 AI Guide 和 Privacy Rules。

关闭笔记本透明打开

索引、搜索和写入前可以临时打开关闭的笔记本，完成后恢复，不要求用户手动整理状态。

最重要的产品边界

这是 MCP-first 的 AI 适配层，不是思源插件，也不是通用 SDK。高风险思源 API 不直接暴露；写入必须经过明确用户意图、confirmed=true、权限检查和快照。

8 个 MCP 工具

工具面刻意压缩成 AI 容易选择的几个动作：启动、刷新、列出、搜索、阅读、创建、编辑、管理。

siyuan_start

会话入口：确保系统笔记本，解析隐私规则，刷新安全索引，返回启动包。

系统维护先调用

siyuan_refresh_index

显式刷新安全索引。会话中途 refresh 不清理 ai_workspace。

索引不清理 workspace

siyuan_list

列出可见笔记本或某路径下一层文档，依赖本地安全索引。

只读分页

siyuan_find

通过思源搜索 API 或 SQL 搜索，返回前统一做隐私过滤。

只读API-only 搜索

siyuan_read

按展示块窗口阅读文档；编辑前开启 include_block_ids 获取坐标。

只读block window

siyuan_create

创建或重写文档。优先完整可读路径，写入后等待路径同步并刷新索引。

写思源confirmed快照

siyuan_edit

基于引用阅读坐标编辑块、范围或普通 Markdown 表格。

写思源坐标校验快照

安全边界

这部分决定 AI 什么都看不到、什么时候只能读、什么时候能写。

默认只读

read first

list/search/read 都必须应用 Privacy Rules；SQL 模式也不能绕过隐私过滤。

写入闸门

snapshot

写入前必须有明确用户意图、confirmed=true、权限为 read_write，并成功创建思源快照。

Privacy Rules 硬隔离

不可见

隐私规则文档只能由 MCP server 内部读取解析，AI 不可读取、搜索、总结或编辑。

Permission = hidden

不进入索引、列表、搜索、阅读和写入。copy/export 也不允许。

Permission = read_only

可以 list/search/read，也可以 copy/export；不能 create/edit/rename/move/delete。

Permission = read_write

可读可写，但写入仍然必须经过 confirmed=true 和快照。

运行时文件

这几个目录最容易被误认为源代码。实际上它们多数是会话产物或缓存。

knowledge_base/

tree.md 给人和 AI 看客观树；docs.jsonl 给工具解析路径和元数据；notebooks.json 存可见笔记本；privacy_rules.json 是解析缓存。

ai_workspace/

README.md 确保存在；attachments/<doc-id>/assets/ 存阅读时提取的附件；exports/ 存导出的 Markdown。

已知债务

这些是架构文档中已经确认的现状，不是猜测。开发时优先对齐这些风险。

安全一致性

系统笔记本不能隐藏的承诺尚未完全由代码强制；Privacy Rules 目前按 hpath 名称硬隔离，可能误挡非系统同名文档；写入后的部分自动 refresh 路径曾存在系统上下文参数遗漏风险。

模块体积

mcp_server.py 约 2808 行，承担协议、schema、搜索、阅读、写入、表格、附件和文档管理。长期应按领域拆分，测试也要同步模块化。

旧入口漂移

cli.py start 仍引用旧启动包文件；部分旧文档可能保留 refresh 清理 workspace 或旧编辑模型的表述。

发布材料

安装指南和插件 manifest 存在版本、链接和描述漂移；发布相关变更必须跑打包检查，并按规则做外部 Agent 验证。

格式选择

比 HTML 更简洁的方案有，但“人类友好 + 信息完整 + 双击可看”这三项很难同时超过单文件 HTML。

Mermaid

最简洁

适合放进 Markdown 或思源文档，但一张图放不下工具矩阵、债务清单和目录说明。

SVG

最轻

单图最干净，适合嵌入 README；缺点是维护文字和响应式布局比较痛。

HTML

推荐

可以同时放图、卡片、表格、折叠说明和导航，不需要构建，也不依赖网络。
